# Auto-Posting System Design

## Overview

When a video is approved on the Luminous Will dashboard, automatically publish it to YouTube, TikTok, Instagram, and Facebook using their respective APIs. The platform-specific captions, hashtags, titles, and descriptions are already generated by the Python pipeline's metadata generator.

**Goal:** One-tap approve on the dashboard → video goes live on all selected platforms. Connect once per platform, works forever.

**Architecture:** All posting logic runs in Next.js serverless functions on Vercel. OAuth tokens stored in Vercel Blob. URL-based uploads for TikTok/Instagram/Facebook (near-instant). Chunked resumable upload for YouTube (handles any file size within 60s function timeout). All platforms post in parallel.

**Tech Stack:** Next.js 15, Vercel Blob (`@vercel/blob`), YouTube Data API v3, TikTok Content Posting API, Instagram Graph API (Reels), Facebook Pages API.

## Global Constraints

- Next.js 15 + React 19 + Tailwind CSS 4
- Vercel Hobby plan (60s function timeout)
- Dark premium styling: #000 background, #1a1a1a panels, #333 borders, #E8A817 amber accent
- Heavy `//` comments throughout (learning codebase)
- All code in the `luminous-will-web` repo — no changes to the Python pipeline
- Tokens stored in Vercel Blob (not env vars, not database)
- The pipeline's queue entry already contains: `video_url`, `thumbnail_url`, `captions` (per-platform), `script_text`, `duration`, `format`, `topic`

## Section 1: Architecture

The system adds three components to the existing web app:

### Platform Publishers (`lib/publishers/`)
One file per platform. Each publisher:
- Takes a queue entry + its platform's captions
- Uploads the video (URL-based or chunked)
- Sets title, description, hashtags, tags per platform
- Returns a `PostResult`: `{ success, url?, error? }`

### Post Orchestrator (`lib/publisher.ts`)
Called by the approve API route. Receives the queue entry and list of selected platforms. Runs all selected publishers in parallel. Updates the queue entry with per-platform results.

### Settings Page (`/settings`)
Four platform cards with OAuth connect/disconnect buttons. Tokens saved to Blob on callback. Status shown per platform.

### Data Flow
```
Dashboard: Approve button (with platform toggles)
  → POST /api/queue/[id]/publish { platforms: ["youtube", "tiktok", ...] }
  → Post orchestrator
    → YouTube publisher (chunked upload + metadata)     ─┐
    → TikTok publisher (URL-based + caption)             ├─ in parallel
    → Instagram publisher (URL-based + caption)          │
    → Facebook publisher (URL-based + description)      ─┘
  → Update queue entry: status="posted", post_results={...}
  → Response to dashboard with results
```

## Section 2: OAuth & Token Storage

### Token Storage
Refresh tokens stored in Vercel Blob as individual JSON files:
- `tokens/youtube.json`
- `tokens/tiktok.json`
- `tokens/meta.json` (covers both Instagram and Facebook)

Each file contains:
```json
{
  "refresh_token": "...",
  "access_token": "...",
  "expires_at": 1719300000,
  "account_name": "Luminous Will"
}
```

### Token Management Module (`lib/tokens.ts`)
Functions:
- `getToken(platform)` — reads token file from Blob, auto-refreshes if expired, saves new access token back
- `saveToken(platform, tokenData)` — writes token file to Blob
- `deleteToken(platform)` — removes token file from Blob
- `isConnected(platform)` — checks if token file exists
- `getConnectionStatus()` — returns status for all platforms

### Auto-Refresh Logic
Before each post, the publisher calls `getToken(platform)`. If `expires_at` is in the past:
1. Use refresh token to request new access token from the platform's token endpoint
2. Save new `access_token` + `expires_at` to Blob
3. Return the fresh access token

If the refresh token itself is rejected (user revoked access, password changed):
1. Delete the token file
2. Return `null` — publisher skips this platform
3. Settings page shows "Needs reconnect" with a yellow warning

### OAuth Flow (same pattern for all platforms)
1. User clicks "Connect [Platform]" on `/settings`
2. Redirect to platform's OAuth consent screen with required scopes
3. User logs in and approves
4. Platform redirects to `/api/auth/[platform]/callback` with auth code
5. Callback exchanges auth code for access token + refresh token
6. Tokens saved to Blob via `saveToken()`
7. Redirect back to `/settings` with success status

### Platform-Specific OAuth Details

**YouTube (Google OAuth 2.0):**
- Scopes: `youtube.upload`, `youtube.readonly`
- Token endpoint: `https://oauth2.googleapis.com/token`
- Requires Google Cloud project with YouTube Data API v3 enabled

**TikTok:**
- Scopes: `video.upload`, `video.publish`
- Token endpoint: `https://open.tiktokapis.com/v2/oauth/token/`
- Requires TikTok Developer app

**Meta (Instagram + Facebook):**
- Scopes: `pages_manage_posts`, `pages_read_engagement`, `instagram_basic`, `instagram_content_publish`
- Token endpoint: `https://graph.facebook.com/v21.0/oauth/access_token`
- Single OAuth flow covers both platforms
- Requires Meta Developer app with Instagram Graph API enabled
- Requires Facebook Page + linked Instagram Business account

## Section 3: Platform Publishers

### Shared Interface (`lib/publishers/types.ts`)

```typescript
interface PostResult {
  platform: string;
  success: boolean;
  url?: string;         // link to the published post
  error?: string;       // human-readable error message
  posted_at?: string;   // ISO timestamp
}

interface PublishInput {
  video_url: string;           // Vercel Blob URL
  thumbnail_url?: string;      // Vercel Blob URL
  captions: {
    caption?: string;
    description?: string;
    title?: string;
    hashtags?: string[];
    tags?: string[];
    category?: string;
  };
  format: "short" | "long";
  topic: string;
  duration?: number;
}
```

### YouTube Publisher (`lib/publishers/youtube.ts`)
- **Upload method:** Resumable chunked upload (5MB chunks)
- **Steps:**
  1. Call `getToken("youtube")` — auto-refreshes if needed
  2. Start resumable upload session: `POST https://www.googleapis.com/upload/youtube/v3/videos?uploadType=resumable`
  3. Download video from Blob URL into memory/stream
  4. Send in 5MB chunks via PUT to the upload URI
  5. Set snippet: title, description, tags, category from `captions.youtube`
  6. Short-form: append `#Shorts` to title
  7. Long-form: include chapters in description if available
  8. Upload thumbnail via `POST /youtube/v3/thumbnails/set`
  9. Return `PostResult` with YouTube video URL
- **Privacy:** Public by default

### TikTok Publisher (`lib/publishers/tiktok.ts`)
- **Upload method:** URL-based (TikTok pulls from Blob URL)
- **Steps:**
  1. Call `getToken("tiktok")`
  2. Initialize upload: `POST https://open.tiktokapis.com/v2/post/publish/video/init/`
  3. Pass `video_url` (Blob URL) — TikTok downloads it
  4. Set caption: combine `captions.tiktok.caption` + hashtags as `#tag` strings
  5. Return `PostResult` with TikTok video URL
- **Near-instant** on the server side — TikTok does the heavy lifting

### Instagram Publisher (`lib/publishers/instagram.ts`)
- **Upload method:** URL-based (Meta pulls from Blob URL)
- **Steps:**
  1. Call `getToken("meta")`
  2. Create media container: `POST /{ig-user-id}/media` with `video_url`, `caption`, `media_type=REELS`
  3. Poll container status until ready (Meta processes the video)
  4. Publish: `POST /{ig-user-id}/media_publish` with container ID
  5. Caption includes hashtags appended as `#tag` strings
  6. Return `PostResult` with Instagram post URL

### Facebook Publisher (`lib/publishers/facebook.ts`)
- **Upload method:** URL-based (Meta pulls from Blob URL)
- **Steps:**
  1. Call `getToken("meta")` (same token as Instagram)
  2. Upload video: `POST /{page-id}/videos` with `file_url` (Blob URL)
  3. Set description from `captions.facebook.description`
  4. Append hashtags inline
  5. Return `PostResult` with Facebook post URL

## Section 4: Post Orchestrator (`lib/publisher.ts`)

### Core Function
```typescript
async function publishToplatforms(
  entry: QueueEntry,
  platforms: string[]
): Promise<Record<string, PostResult>>
```

### Behavior
1. **Pre-flight validation** (before any uploads):
   - `video_url` exists and is accessible (HEAD request)
   - Each selected platform has a valid token (via `isConnected()`)
   - Captions exist for each platform (fall back to topic as title if missing)
2. **Post in parallel:** `Promise.allSettled()` on all selected publishers
3. **One retry** per failed platform (network timeout or rate limit only — not auth errors)
4. **Update queue entry:**
   - `post_results`: per-platform `PostResult` objects
   - `status`: `"posted"` if at least one platform succeeded, `"failed"` if all failed
   - `error`: summary message if any failures
5. **Return** full results to the dashboard

### Error Categories
- **Auth error** (token revoked/expired beyond refresh) → skip platform, mark "needs reconnect"
- **Network error** (timeout, DNS) → retry once, then fail with message
- **Rate limit** (429) → wait 2 seconds, retry once
- **API error** (400/403/500 from platform) → fail with platform's error message
- **Video URL inaccessible** → fail all platforms, don't attempt any uploads

## Section 5: Dashboard Changes

### Approve Flow
Current: single "Approve" button → calls `/api/queue/[id]/approve` → sets status to `"approved"`.

New: The existing approve route is no longer called directly. Instead:
1. Click "Approve" → panel expands showing:
   - Four platform toggles (YouTube, TikTok, Instagram, Facebook) — all ON by default
   - Disconnected platforms shown as greyed out with "Connect in Settings" hint
   - Optional date/time picker for scheduling (blank = immediate)
2. Click "Post Now" → calls `/api/queue/[id]/publish` with `{ platforms: [...] }` for immediate posting
   Click "Schedule" → calls `/api/queue/[id]/approve` with `{ scheduled_post_time: "..." , target_platforms: [...] }` for deferred posting
3. Dashboard shows posting spinner per platform
4. Results appear: green check + link for success, red X + error for failure
5. "Retry Failed" button appears if any platform failed

### Post Results Display
After posting, each queue entry shows a results section:
- Per-platform row: platform icon + status + clickable post URL or error message
- Timestamp of when it was posted

### Scheduled Posts Display
- Clock icon with scheduled date/time
- "Cancel" button to revert to `pending_review`

### Navigation
- Add "Settings" link in dashboard header/nav for easy access to platform connections

### Status Flow
```
pending_review → posting → posted (success)
                         → failed (all platforms failed)
pending_review → approved (scheduled — waiting for cron)
               → posting → posted/failed
```

## Section 7: Scheduled Posts Cron

### Vercel Cron Configuration
In `vercel.json`:
```json
{
  "crons": [{
    "path": "/api/cron/post-scheduled",
    "schedule": "*/5 * * * *"
  }]
}
```

### Cron Route (`/api/cron/post-scheduled`)
1. Verify request is from Vercel cron (check `CRON_SECRET` header)
2. Load queue from Blob
3. Find entries where `status === "approved"` AND `scheduled_post_time` is in the past
4. For each: call the post orchestrator with `target_platforms`
5. Update queue entries with results

## Section 9: Reliability Guarantees

### Pre-Post Validation
- Token exists and is valid (auto-refresh if expired)
- Video URL is accessible (HEAD request before upload)
- Captions exist for selected platforms (fallback to topic if missing)
- At least one platform is connected and selected

### During Posting
- All platforms post in parallel (`Promise.allSettled`)
- Each platform is independent — one failure never blocks others
- One automatic retry on network timeout or rate limit (not on auth errors)
- Every error stored as clear human-readable message in `post_results`

### After Posting
- Per-platform post URLs stored in queue entry for verification
- Status reflects actual outcome: `"posted"` (at least one success), `"failed"` (all failed)
- "Retry Failed" button retries only the platforms that failed, not all
- Dashboard shows exactly what happened per platform

### Token Lifecycle
- Auto-refresh before every post attempt
- If refresh fails → mark platform as "needs reconnect"
- Settings page shows which platforms need attention
- Reconnect is one click — same OAuth flow as initial connect
